Two-Page Solution Brief

Use the brief as the technical evaluation asset after landing-page engagement.

Page 1

Executive and business context

Citadel's advantage is built on turning data into insight, models, and trading strategy faster and more reliably than the market. For selected GCP workloads, the challenge is how to provide high-performance shared file access close to cloud compute without adding unnecessary Lustre operations burden.

  • Challenge: determine where Managed Lustre is the right tool and where it is not.
  • Recommended solution: evaluate workloads that run in GCP, require shared file access, and have measurable criteria.
  • Use cases: checkpointing, distributed AI training, GKE workflows, cloud HPC, simulation, and burst research.
Page 2

Technical fit, proof plan, and next step

Managed Lustre may sit as an active performance file system alongside GCP compute, GKE, Cloud Storage, on-prem performance systems, and Citadel's monitoring, access control, security, and automation frameworks.

  • Checklist: GCP execution, POSIX file access, checkpointing, multi-node compute, measurable baselines, named owner.
  • Metrics: checkpoint duration, restore time, data-loader throughput, compute wait time, concurrency, metadata, operations, cloud movement.
  • Next step: focused Google + DDN AI/HPC workload fit assessment.
Metric Why It Matters How To Measure
Checkpoint duration

Long checkpoints slow model iteration.

Compare current checkpoint time to target workload requirements.

Restore time

Faster restart improves experimentation resilience.

Measure time to resume from saved state.

Data-loader throughput

Training productivity depends on feeding compute consistently.

Measure data ingestion behavior during representative jobs.

Compute wait time

Idle GPUs or TPUs reduce infrastructure value.

Capture wait states attributable to data access.

Operational effort

Managed service value depends on reduced admin burden.

Compare setup, maintenance, monitoring, and support effort.

We are running a focused Google + DDN campaign around Managed Lustre for GCP-based AI and HPC workloads. For Citadel, we are not assuming broad fit. We are looking for one workload where high-performance shared file access could matter.

Based on the account plan, Citadel's business value comes from research velocity, AI productivity, and infrastructure control. If a GCP workload is slowed by checkpointing, data movement, or active file access, that may be worth validating.

BDR Talk Track

This is not a generic storage campaign. The ask is modest: identify whether there is one GCP-based AI, checkpointing, simulation, or research workload where managed high-performance file access is worth evaluating with Google and DDN.

SE / Technical Talk Track

Map the workload, data path, compute environment, concurrency, checkpoint behavior, file semantics, Cloud Storage interaction, GKE or Compute Engine model, security requirements, and current baseline metrics.

Partner Talk Track

Google and DDN can help Citadel evaluate whether Managed Lustre is the right GCP-native performance file system for a specific AI, HPC, checkpointing, simulation, or research workload.

Objection Handling
We already have high-performance storage.
The question is whether any GCP-based workload needs managed high-performance shared file access close to cloud compute.
We can run Lustre ourselves.
Citadel likely can. The evaluation is whether self-managing Lustre is the best operating model for a specific GCP workload.
Object storage is our cloud standard.
Object storage may remain the durable tier. Managed Lustre is for active AI/HPC file workflows.
We need proof, not claims.
Agreed. The recommended next step is a workload fit assessment with baseline metrics and success criteria.
Discovery Questions
  • Where does GCP fit in Citadel's AI/HPC or research platform roadmap?
  • Which GCP workloads require shared file access rather than object-only access?
  • Are checkpoint-heavy workloads running or being evaluated in GCP?
  • What concurrency, metadata, and throughput patterns should be tested?
  • What cost model compares cloud execution against on-prem execution?
  • Does Google confirm relevant AI/HPC or research workloads in GCP?
Claims Guardrails
  • Approved: Managed Lustre is designed for AI and HPC workloads in Google Cloud.
  • Use caution: performance improvement only when workload fit is validated.
  • Do not claim guaranteed faster training or guaranteed GPU utilization improvement.
  • Claims needing validation: performance, throughput, cost savings, security, competitive comparisons, and Google partner language.

Opening Problem

Citadel Securities does not evaluate infrastructure in isolation. Every data platform decision has to connect to research velocity, compute productivity, operational control, or trading infrastructure resilience. For cloud-based AI and HPC workloads, compute can often be provisioned faster than the data layer can keep up.

Why Now

AI and HPC workloads are moving from experimentation toward repeatable production patterns. Managed Lustre is relevant only where workload fit is real: GCP-based AI training, checkpointing, simulation, GKE-based pipelines, or burst research environments that require high-throughput shared file access.

GKE-based AI workloads

Cloud HPC / simulation

Burst research environments

Solution Section

Google Cloud Managed Lustre powered by DDN is a managed high-performance parallel file system for AI and HPC workloads in Google Cloud. For Citadel, its value should be tested where active file performance affects research execution: checkpointing, distributed training, simulation, market replay, and cloud-adjacent research workflows.

How It Works

  1. Identify the workload: select one GCP-based or GCP-candidate workload.
  2. Define the current-state baseline: capture runtime, checkpoint duration, restore time, data movement, concurrency, operations, and compute wait indicators.
  3. Validate the architecture: review GCP compute, GKE, Cloud Storage, networking, identity, security, and Citadel's operating model.
  4. Decide based on proof: proceed only if workload fit is credible.

Proof / Validation

  • Whether the workload truly requires high-performance shared file access.
  • Current checkpoint and restore times.
  • Current data-loader or file-access bottlenecks.
  • GPU/TPU wait time related to data access.
  • GKE or Compute Engine deployment model.
  • Cloud Storage staging and transfer requirements.
  • Operational fit with Citadel monitoring, access control, security, and automation.

When should it be considered instead of Cloud Storage?

When active workloads need shared file semantics, high throughput, low latency, and parallel access close to cloud compute.

Is this only for AI training?

No. It may also apply to simulation, research burst, GKE-based AI workflows, and HPC-style workloads in GCP.
